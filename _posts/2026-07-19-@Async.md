`@Async`가 AOP로 동작하고, 그로 인해 내부 호출(self-invocation)에서는 동작하지 않는다는 사실은 많이 알려져 있다.

실제로 내부에서 어떤 흐름으로 처리되는지 궁금해서 소스 코드를 보며 정리해봤다.


## @Async

- `@Async`는 메서드 또는 클래스에 붙일 수 있는 애노테이션이다. 
- `value`로 사용할 Executor의 이름(qualifier)을 지정할 수 있으며, 생략하면 기본 Executor가 사용된다.

```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Reflective
public @interface Async {
    String value() default "";
}
```

## AnnotationAsyncExecutionInterceptor

`@Async`가 붙은 메서드가 호출되면 AOP 프록시가 이 인터셉터를 통해 가로챈다.

클래스 계층은 아래와 같다.

```
AnnotationAsyncExecutionInterceptor
    └─ AsyncExecutionInterceptor        (invoke 로직 담당)
        └─ AsyncExecutionAspectSupport  (executor 결정, task 제출, 에러 처리)
```

### AsyncExecutionInterceptor.invoke()

- AOP가 메서드 호출을 가로채면 가장 먼저 실행되는 진입점이다. 
- 실제 메서드 실행을 `Callable`로 감싼 뒤 executor에 제출한다.

```java
@Override
@Nullable
public Object invoke(final MethodInvocation invocation) throws Throwable {
    // 1. 실행할 메서드 결정
    Class<?> targetClass = (invocation.getThis() != null ? AopUtils.getTargetClass(invocation.getThis()) : null);
    Method specificMethod = ClassUtils.getMostSpecificMethod(invocation.getMethod(), targetClass);
    final Method userDeclaredMethod = BridgeMethodResolver.findBridgedMethod(specificMethod);

    // 2. executor 결정
    AsyncTaskExecutor executor = determineAsyncExecutor(userDeclaredMethod);
    if (executor == null) {
        throw new IllegalStateException(
                "No executor specified and no default executor set on AsyncExecutionInterceptor either");
    }

    // 3. 실제 메서드 호출을 Callable로 래핑
    Callable<Object> task = () -> {
        try {
            Object result = invocation.proceed();
            if (result instanceof Future<?> future) {
                return future.get();
            }
        }
        catch (ExecutionException ex) {
            handleError(ex.getCause(), userDeclaredMethod, invocation.getArguments());
        }
        catch (Throwable ex) {
            handleError(ex, userDeclaredMethod, invocation.getArguments());
        }
        return null;
    };

    // 4. executor에 제출
    return doSubmit(task, executor, invocation.getMethod().getReturnType());
}
```

### AsyncExecutionAspectSupport.determineAsyncExecutor()

- `@Async(value = "executorName")`으로 지정된 qualifier를 보고 그에 맞는 Executor를 찾는다. 
- 같은 메서드에 대한 조회는 캐싱되어 반복 탐색 비용을 줄인다.

```java
@Nullable
protected AsyncTaskExecutor determineAsyncExecutor(Method method) {
    // 1. 캐시 확인
    AsyncTaskExecutor executor = this.executors.get(method);
    if (executor == null) {
        Executor targetExecutor;
        // 2. @Async value 속성에서 qualifier 조회
        String qualifier = getExecutorQualifier(method);
        if (this.embeddedValueResolver != null && StringUtils.hasLength(qualifier)) {
            qualifier = this.embeddedValueResolver.resolveStringValue(qualifier);
        }
        // 3. qualifier가 있으면 해당 executor, 
        if (StringUtils.hasLength(qualifier)) {
            targetExecutor = findQualifiedExecutor(this.beanFactory, qualifier);
        }
        
        // 4. 없으면 default executor
        else {
            targetExecutor = this.defaultExecutor.get();
        }
        if (targetExecutor == null) {
            return null;
        }
        executor = (targetExecutor instanceof AsyncTaskExecutor asyncTaskExecutor ?
                asyncTaskExecutor : new TaskExecutorAdapter(targetExecutor));
        // 5. 캐시 저장
        this.executors.put(method, executor);
    }
    return executor;
}
```

#### AnnotationAsyncExecutionInterceptor.getExecutorQualifier()

- qualifier를 찾는 우선순위가 있다. 메서드에 붙은 `@Async`를 먼저 확인하고, 없으면 클래스에 붙은 `@Async`를 확인한다.

```java
@Override
@Nullable
protected String getExecutorQualifier(Method method) {
    // 1. 메서드에서 @Async 조회
    Async async = AnnotatedElementUtils.findMergedAnnotation(method, Async.class);
    if (async == null) {
        // 2. 없으면 클래스에서 @Async 조회
        async = AnnotatedElementUtils.findMergedAnnotation(method.getDeclaringClass(), Async.class);
    }
    return (async != null ? async.value() : null);
}
```

### AsyncExecutionAspectSupport.doSubmit()

- 반환 타입에 따라 task를 다른 방식으로 executor에 제출한다. 
- `void` 메서드는 결과 없이 제출만 하고, `Future` 계열은 호출자가 결과를 받을 수 있도록 반환한다.

```java
@Nullable
@SuppressWarnings("deprecation")
protected Object doSubmit(Callable<Object> task, AsyncTaskExecutor executor, Class<?> returnType) {
    if (CompletableFuture.class.isAssignableFrom(returnType)) {
        return executor.submitCompletable(task);
    }
    else if (org.springframework.util.concurrent.ListenableFuture.class.isAssignableFrom(returnType)) {
        return ((org.springframework.core.task.AsyncListenableTaskExecutor) executor).submitListenable(task);
    }
    else if (Future.class.isAssignableFrom(returnType)) {
        return executor.submit(task);
    }
    else if (void.class == returnType) {
        executor.submit(task);
        return null;
    }
    else {
        throw new IllegalArgumentException(
                "Invalid return type for async method (only Future and void supported): " + returnType);
    }
}
```

### AsyncExecutionAspectSupport.handleError()

- 비동기 작업 중 예외가 발생했을 때의 처리 방식은 반환 타입에 따라 달라진다.
    - **`Future` 계열** : 예외를 다시 던져 `Future.get()` 호출 시 `ExecutionException`으로 받을 수 있게 한다.
    - **`void`** : 호출자에게 예외를 전달할 방법이 없으므로 `AsyncUncaughtExceptionHandler`로 위임한다.
- 기본 구현체인 `SimpleAsyncUncaughtExceptionHandler`는 ERROR 레벨 로깅만 하고 예외를 삼킨다. 
- 즉, **`void` 메서드에서 발생한 예외는 별도 처리를 하지 않으면 error 로그만 남는다.**

```java
protected void handleError(Throwable ex, Method method, Object... params) throws Exception {
    if (Future.class.isAssignableFrom(method.getReturnType())) {
        ReflectionUtils.rethrowException(ex);
    }
    else {
        try {
            this.exceptionHandler.obtain().handleUncaughtException(ex, method, params);
        }
        catch (Throwable ex2) {
            logger.warn("Exception handler for async method '" + method.toGenericString() +
                    "' threw unexpected exception itself", ex2);
        }
    }
}
```
